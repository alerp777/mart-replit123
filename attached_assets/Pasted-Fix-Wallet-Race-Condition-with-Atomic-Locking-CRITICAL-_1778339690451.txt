What's broken: Wallet balance updates are not atomic. This sequence can happen:

User A has 100 PKR
Request 1: Read balance (100) → check >= 50 ✓ → PAUSE
Request 2: Read balance (100) → check >= 90 ✓ → UPDATE to 10 ✓
Request 1: Resume → UPDATE to 50 ✓
Final balance: 50 (but should be negative! User was double-charged)
This is a CRITICAL race condition that causes double-spend bugs and financial loss.

What to do:

Rewrite all wallet deduction logic to use a SINGLE atomic transaction with row-level locking
Use SELECT ... FOR UPDATE to lock the wallet row
All balance checks and updates happen in a single transaction (no interleaving possible)
Use Drizzle db.transaction() with explicit SQL for the lock
Database pattern (SQL):

SQL
BEGIN;
SELECT balance FROM wallets WHERE user_id = $1 FOR UPDATE;
-- App checks: if balance >= amount
UPDATE wallets SET balance = balance - $amount WHERE user_id = $1;
COMMIT;

// services/wallet.ts
export async function deductWalletBalance(
  userId: string,
  amount: number,
  reason: string
) {
  return await db.transaction(async (tx) => {
    // 1. Lock the wallet row
    const walletLocked = await tx.execute(sql`
      SELECT ${wallets.balance} FROM ${wallets} 
      WHERE ${eq(wallets.userId, userId)}
      FOR UPDATE
    `);
    
    if (!walletLocked.length) {
      throw new Error('Wallet not found');
    }
    
    const currentBalance = walletLocked[0].balance;
    
    // 2. Atomic check + update
    if (currentBalance < amount) {
      throw new Error('Insufficient balance');
    }
    
    // 3. Deduct (while row is locked)
    const result = await tx.update(wallets)
      .set({ balance: sql`${wallets.balance} - ${amount}` })
      .where(eq(wallets.userId, userId))
      .returning({ newBalance: wallets.balance });
    
    // 4. Audit log
    await tx.insert(walletAuditLogs).values({
      userId,
      type: 'DEDUCTION',
      amount,
      reason,
      newBalance: result[0].newBalance,
      timestamp: new Date()
    });
    
    return result[0];
  });
}
